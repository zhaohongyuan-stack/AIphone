## 5. ⚠️ 联调冲突问题清单（核心，需本地处理）

以下为两服务同机联调时**必然**或**可能**出现的问题，按严重程度排序：

### 问题 1【严重】Redis 端口不一致 → 两服务状态不互通
- Java：`localhost:6380`
- Python：`localhost:6379`
- 影响：坐席状态、排队队列、事件去重标记等 Redis 共享数据完全隔离，联调时 Java 看不到 Python 写的状态，反之亦然。
- 处理建议：统一到 `6380`（Docker Redis）。修改 `IVR/.env`：`REDIS_PORT=6380`。

### 问题 2【严重】RocketMQ 消费组冲突 → CCC 事件被随机分流
- Java local profile：group `GID_CCC_TEST_CONSUMER`，topic `testTopic`
- Python .env：group `GID_CCC_TEST_CONSUMER`，topic `testTopic`（**同一 group + 同一 topic**）
- 但两者 RocketMQ endpoint **不同**（Java `rmq-cn-vo14vdr3p01`，Python `rmq-cn-1tb4v0w5v05`）。
- 影响：若 endpoint 不一致，各消费各的实例，不冲突；但若本地把两者 endpoint 统一（建议统一到 Java 用的 `rmq-cn-vo14vdr3p01`），则同一 group 下两消费者会**负载均衡**瓜分 CCC 事件，导致事件随机进入 Java 或 Python，行为不可预期。
- 处理建议：明确两服务职责边界——CCC 事件应由哪一端消费？若都消费，必须用不同 group；若只一端消费，另一端关闭 `mq.enabled`。需与架构确认。

### 问题 3【严重】PostgreSQL 数据源不一致 → 数据不共享
- Java 默认/local：本地 Docker `localhost:5433/ai_phone` (postgres/123456)
- Python .env：云 RDS `pgm-...:5432/ai_phone` (supabase_admin/AI_phone123)
- 影响：两服务操作的是不同的 `ai_phone` 库，工单/坐席数据互不可见。
- 处理建议（取决于联调目标）：
  - 方案 A（纯本地联调，推荐）：Python 也指向本地 Docker PG。改 `IVR/.env`：
    ```
    PG_HOST=localhost
    PG_PORT=5433
    PG_USER=postgres
    PG_PASSWORD=123456
    PG_DATABASE=ai_phone
    ```
    注意：需在本地 Docker PG 额外建 `robot_agent_db` 库并赋权（见问题 4）。同时注意两服务的表是否会重名/冲突（Java 用 schema.sql 建 ai_phone 表，Python 用 SQLAlchemy 建 work_order/dialogue_detail/agent_info 等），建议联调前核对表名。
  - 方案 B（Python 保留云 RDS 真实数据）：Python 不改，但需本机能访问云 RDS（公网 + 白名单）。此时 Java 仍用本地空库，两服务数据依旧不共享，仅适合接口级联调。

### 问题 4【中】robot_agent_db 在本地 Docker PG 中不存在
- Python 依赖 `robot_agent_db` 库（智能坐席状态），`.env` 指向 `localhost:5432`（postgres/456789）。
- 本地 Docker PG（compose.yaml）只建了 `ai_phone`，端口 5433，账号 postgres/123456。
- 影响：若按问题 3 方案 A 统一到本地 Docker，Python 启动会连不上 robot_agent_db，降级为「仅 Redis 模式」。
- 处理建议：在本地 Docker PG 里手动建库并改 .env：
  ```sql
  -- 连接 5433 后执行
  CREATE DATABASE robot_agent_db;
  ```
  并把 `IVR/.env` 的 `ROBOT_DB_HOST=localhost`、`ROBOT_DB_PORT=5433`、`ROBOT_DB_USER=postgres`、`ROBOT_DB_PASSWORD=123456`。启动时 `init_robot_db.py` 会自动建表 + 执行 `tests/robot_agent_status.sql`。

### 问题 5【中】KB_EXCEL_PATH 路径错误
- 当前 `d:\IVR\副本知识库导出记录-20260629.xlsx`（旧 Windows 路径）
- 处理建议：改为 `C:\code\Aiphone\IVR\副本知识库导出记录-20260629.xlsx`（文件已在包内）。

### 问题 6【低】Java schema.sql 每次启动执行
- 默认 profile `spring.sql.init.mode: always`，每次启动都跑 schema.sql。
- 影响：若本地 PG 已有数据，重复执行可能报错或重置。
- 处理建议：本地联调首次启动建表后，把 `application.yml` 改为 `mode: never`，或用 local profile 覆盖。

### 问题 7【低】DashScope API Key 未注入 Java
- Java 默认/local profile 的 `DASHSCOPE_API_KEY` 为空（ASR 不可用）。
- 处理建议：需要 ASR 时，在 `application-local.yml` 补 `aliyun.voice.api-key: sk-8542...`（值见 IVR/.env）。

### 问题 8【低】RabbitMQ 在 Java 默认配置中会尝试连接
- application.yml 配了 RabbitMQ，pom 也有 spring-boot-starter-amqp。若不排除且中间件未起，启动会失败/重试。
- 处理建议：本地用 compose.yaml 起了 RabbitMQ:5673，可正常连；若不用，参考 prod 的 `SPRING_AUTOCONFIGURE_EXCLUDE` 排除。
